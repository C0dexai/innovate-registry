Workflow in Action: Manifest Execution

Orchestration is defined in human-readable YAML manifests. The CUA Engine reads these manifests and executes the steps in sequence, passing context between services.

Deploy-WebApp-v1.mcp.yaml

services: - name: ImageMaker - name: CUA_Engine - name: GitHub_CI_CD workflow: - step: 1 name: "Generate UI Assets" service: ImageMaker payload: prompt: "Create a logo for 'Project Phoenix'." publish_as: "ui_assets_result" - step: 2 name: "Scaffold Application Code" service: CUA_Engine payload: prompt: "Using UI assets from {{ui_assets_result}}, scaffold a React frontend..." publish_as: "scaffold_result" - step: 3 name: "Trigger CI/CD Pipeline" service: GitHub_CI_CD payload: inputs: source_code_ref: "{{scaffold_result.code_path}}"
1

Generate UI Assets

The ImageMaker service is called to create a logo. The result is published to the context as `ui_assets_result`.

2

Scaffold Application Code

The CUA Engine consumes the `ui_assets_result` and instructs agent KARA to generate the application's source code.

3

Trigger CI/CD Pipeline

The generated code path is passed to the GitHub CI/CD service, which triggers an automated build, test, and deployment workflow.

Standardized API & Feedback Loop

Standard Service Endpoints

Every application in the ecosystem adheres to a standard API contract, ensuring predictable and reliable interoperability.

POST /task - Submit a new job
GET /task/{taskId} - Check job status
GET /result/{taskId} - Retrieve job output

Real-Time Feedback Architecture

A Pub/Sub message broker, managed by agent GUAC, provides real-time status updates across the entire system.
